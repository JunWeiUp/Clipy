## 根因（已确认）

截图后从 200MB 只回落到 160MB（残留 ~85MB），是**多因素叠加**，按贡献排序：

1. **`ScreenshotImageProcessor.sharedCIContext` 的 GPU/IOSurface 纹理池不收缩**（`ScreenshotImageProcessor.swift:6`）。仅在用户画**马赛克标注**时触发（`AnnotationCanvasView.drawFlattenedMosaic:777`、`applyPixellate:346`）。CIContext 处理过大图后，内部 IOSurface pool 按最大尺寸驻留整个进程生命周期，进程结束前不收缩。

2. **AppKit layer backing store / WindowServer IOSurface 延迟回收**：4 个 `backing: .buffered` panel（`CaptureOverlayWindow.swift:461` overlay 全屏、`CaptureAnnotationPanel.swift:16`、magnifier、toolbar）。`close()`/`orderOut` 后 AppKit 不保证立即把 IOSurface 还给系统。

3. **malloc zone 高水位不收缩（最普遍，无关标注）**：截图峰值同时存在 6-7 份大位图（整屏抓图 33MB + 裁剪 + NSImage 包装 + flatten CGContext + PNG 编码 + pasteboard）。即使 `autoreleasepool` 回收对象，libmalloc 扩张的虚拟/物理页不立即还给系统 → RSS 不回落。这是 macOS 的正常行为，但表现为"内存没降"。

4. **现有 `MemoryFootprintReclaimer` 完全无效**（`MemoryFootprintReclaimer.swift:22-25`）：只清 `recentSummaries`（<100KB 字符串），不碰 CIContext/layer/pasteboard。这是"已有机制为何没解决"的直接原因。

## 修复方案（深度优化，按你的选择）

### A. CIContext 按需创建 + 用后释放（修复大头 #1）

**`ScreenshotImageProcessor.swift`**
- `static let sharedCIContext` → `private(set) static var __ciContext: CIContext?` + 计算属性 `static var sharedCIContext: CIContext`（getter 里 lazy 创建）。
- 新增 `static func releaseCIContext()`：置 nil。下次马赛克标注时按需重建。
- CIContext 创建成本低（几十 ms），释放后能压几十 MB IOSurface 缓存，值得。

**`ScreenshotCoordinator.swift`**
- `handleCaptureResult`（截图流程唯一收尾点，fullscreen/region/window 都会调）末尾调 `ScreenshotImageProcessor.releaseCIContext()`。

### B. 截图后主动触发 malloc zone 复位（修复大头 #3）

**新增 `MemoryFootprintReclaimer.reclaimAfterScreenshot()`**：
- 遍历所有 malloc zone，对每个 zone 调 `malloc_zone_malloc_pressure_event(zone, 0)`（macOS 10.x+ libmalloc API，触发 zone 整理并尝试 madvise 归还页给系统）。这是 Apple 推荐的"释放已知峰值内存"手段。
- 实现用 `malloc_default_zone()` + `malloc_zone_enumerate`，或更简单地遍历 `malloc_zone_for_ptr` / 公开 API。
- 保险起见用 `malloc_zone_malloc_pressure_event`（在 `<malloc/malloc.h>`），失败则静默跳过。
- 触发时机：`ScreenshotCoordinator.handleCaptureResult` 末尾，`dispatch_async` 到后台队列执行（避免阻塞 UI）。

### C. 扩充 `MemoryFootprintReclaimer`（修复 #4）

在 `reclaimIfIdle()` 和新增的 `reclaimAfterScreenshot()` 中，除了原有 `releaseMenuMemory`，增加：
- `ScreenshotImageProcessor.releaseCIContext()`（A 已实现，这里复用）
- malloc zone 复位（B 已实现）

### D. annotation panel 透明底图复用（小优化，修复 #2 的局部）

**`CaptureAnnotationPanel.swift`**
- `transparentImage(size:)` 改为按 size 缓存：`private static var transparentImageCache: [CGSize: NSImage]`（尺寸完全相同时复用，避免拖动选区时反复 `lockFocus` 生成新位图）。cache 在 panel 销毁时清。
- 实际上 `reposition` 已有 `guard frame.size != canvasSize` 守卫，只在尺寸变化时生成——但拖动过程中尺寸会频繁变。缓存命中率取决于拖动行为，收益中等，低风险。

### E. `ScreenshotCaptureService.captureDisplayRegionByCropping` 不再全程持有整屏图（大头 #3 峰值降低）

**`ScreenshotCaptureService.swift:220` `captureDisplayRegionByCropping`**：
- 当前：抓全屏 → `fullImage`（33MB）→ `cropping` → 返回。`fullImage` 在 autoreleasepool 内会被释放，但峰值时刻 `fullImage` + `cropped` 同时存在。
- 优化：crop 后立即把 `fullImage` 置 nil（在 autoreleasepool 内手动断引用），让它在 cropping 完成的同一 runloop 回收，缩短双图并存窗口。
- 注意：`SCScreenshotManager.captureImage` 是同步的，`fullImage` 是局部变量，作用域结束即释放；但若 `cropping(to:)` 的返回值和 `fullImage` 共享 backing（CGImage crop 可能是 lazy view），则不会真正释放。需用 `cropping(to:)` 后立即 `fullImage = nil` 并依赖 autoreleasepool。

## 验证

- `swiftc -typecheck` 全量类型检查通过
- 场景走查：
  1. 普通截图（无标注）→ 完成 → 内存应明显回落（malloc zone 复位 + CIContext 即使没用到也释放）
  2. 带马赛克标注的截图 → 完成 → 内存应回落（CIContext 释放 + malloc zone 复位）
  3. 反复截图 5 次 → 内存不应单调爬升（每次结束都复位）
  4. 截图时无卡顿（malloc zone 复位在后台队列；CIContext 重建在下次马赛克时，几十 ms 可接受）

## 预期效果

- 无标注截图：峰值后能多回落 20-40MB（主要来自 malloc zone madvise）
- 马赛克标注截图：峰值后能多回落 40-80MB（CIContext IOSurface pool 释放 + malloc zone）
- 160MB → 预计可降到 90-110MB，接近 75MB 基线（剩余 15-35MB 是 layer backing + pasteboard + NSImage 历史缩略，属合理常驻）

## 注意

malloc zone 复位对正在使用的内存页无效（它只能 madvise 已释放但未归还的页），所以不可能完全回到 75MB——但能回收大部分"用完未还"的峰值内存，这是 macOS 应用内存管理的标准做法。
