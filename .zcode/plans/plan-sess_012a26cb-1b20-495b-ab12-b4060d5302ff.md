## 目标
只改 macOS 端（`clipy_macos/`），把它从「不透明 + 纯色背景 + emoji 图标」升级为 Apple Liquid Glass 风：透明窗口 + 毛玻璃材质层 + SF Symbol 菜单图标 + 极轻克制动效。Flutter 端不动。

## 设计取向（design-taste-frontend skill 的美学判断，用原生框架落地）
- **Dials：VARIANCE 7 / MOTION 4 / DENSITY 3**（极轻克制）。
- **材质**：原生 SwiftUI `Material`（`.ultraThinMaterial` / `.regularMaterial` / `.thickMaterial`），不是 CSS 伪造。skill Appendix C 明确：web 只是 approximation，原生平台用 HIG 原生材质才是正解。
- **强调色**：单一冷色（系统 `.controlAccentColor` / `.tintColor`），遵守 skill 的「One accent color」+「Color Consistency Lock」。
- **字体**：保持 SF Pro（系统默认），收拢散落的 `.font(.system(size:))` 到 `AppFont`。
- **动效**：MOTION 4 = 仅窗口淡入 + hover 微反馈（`scaleEffect`/`opacity` transition），全部 `transform`/`opacity`，遵守 reduced-motion。

## 改造步骤（从低风险到高风险）

### 1️⃣ 菜单栏图标换 SF Symbol（最低风险，1 处改动）
- 文件 `MenuController.swift:98-100`：`button.title = "📋"` → 用 `NSImage(systemSymbolName: "clipboard", accessibilityDescription:)` 设为 `button.image`，并配 `symbolConfiguration`（`.scale(.medium)`，与系统状态栏图标一致）。
- 备选符号：`clipboard` / `list.clipboard` / `clipboard.fill`。用 `clipboard`。

### 2️⃣ 扩展设计 token（基础设施，不改行为）
- 文件 `DesignTokens.swift`：
  - `AppCornerRadius` 增加 `medium = 8`、`large = 12`（收拢散落的 6/8/12）。
  - `AppColor`（当前在 `AppWindowLayout.swift`）扩展：新增 `accent`（`Color(nsColor: .controlAccentColor)`）、`separator`、`glassTint` 等，但**不改**现有两个颜色名（保持兼容）。
- 文件 `AppWindowLayout.swift`：把 `AppColor` 留原处或迁到 `DesignTokens.swift`（二选一，倾向迁并集中，但要更新所有引用——先评估引用面）。

### 3️⃣ 窗口透明化 + 毛玻璃背景（核心，单一文件 `HostingWindow.swift`）
- `backgroundColor = .windowBackgroundColor` → `.clear`
- `isOpaque = true` → `false`
- 新增 `titlebarAppearsTransparent = true`、`titleVisibility = .hidden`（窗口内容自己承担标题区，避免双层标题）
- 新增 `titlebarSeparatorStyle = .none`
- `contentView` 背景用一个 `NSVisualEffectView`（`material = .underWindowBackground`，`blendingMode = .behindWindow`，`state = .active`）垫在最底层 —— 这是让整个窗口内容呈现毛玻璃的标准做法。
- 关键兼容：`contentViewController = NSHostingController` 模式下，需要把 vibrancy view 作为 hosting controller view 的 background，或用 `contentView` 嵌套。**先小范围验证一种方案**（见步骤 6 试点）。
- 加 `isMovableByWindowBackground = false`（避免误拖）。

### 4️⃣ 布局层切换到 `Material` 背景
- `AppListWindowLayout` / `AppFormWindowLayout` / `AppWindowHeader` / `AppToolbar`：`.background(AppColor.windowBackground)` / `AppColor.windowChrome` → `.background(.regularMaterial)`（或 `.thinMaterial`），让内容层叠在步骤 3 的窗口玻璃之上，形成层次感。
- `Divider()` → 用系统 separator 即可（在材质上会自动半透明）。

### 5️⃣ 极轻克制动效（MOTION 4）
- `HostingWindow.show()`：加 `NSWindow` 的 `alphaValue` 0→1 动画（0.18s，`easeOut`）作为窗口淡入。遵守 reduced-motion（系统「减少动态效果」开启时跳过）。
- `AppToolbarButton` hover：`scaleEffect(1.0)`（保持现状，不过度）—— 仅在自定义悬浮卡片上加 `hover` scale 0.98 反馈。

### 6️⃣ 试点先行（风险隔离）
- **先只对一个窗口验证完整链路**：选 `SearchWindow`（`AppListWindowLayout`，最能体现玻璃效果的大窗口）。
- 验证：透明 + 玻璃 + 标题栏隐藏 + Material 背景在 macOS 13 正常、浅色/深色都正常、窗口可拖动可缩放。
- 试点 OK 后，因为是统一走 `HostingWindow` + 统一 layout，**其余窗口自动继承**，无需逐个改。

## 不做的事（明确边界）
- **不改 Flutter**（用户选择只做 macOS）。
- **不改截图 overlay/pin panel**：它们已是 `.clear` 透明面板 + 部分已用 `NSVisualEffectView(.hudWindow)`，效果已 OK，动了风险高且脱离本次目标。
- **不改菜单 `NSMenu` 本体**：系统渲染，无法自定义玻璃；只改图标和嵌入的自定义 view。
- **不引入新依赖/Swift Package**：纯系统 API。

## 验证
- `proxy` 后跑 `./build_macos_app.sh`，确认 ~75 个 Swift 文件整体编译通过（最可靠的回归检查）。
- 交付时如实报告：编译是否通过；如某窗口视觉异常会说明并回退该处。

## 风险与缓解
- **风险**：`NSHostingController` + `NSVisualEffectView` 嵌套在某些布局下可能出现层级/事件穿透问题。
- **缓解**：步骤 6 先单窗口试点验证，失败则回退该窗口、保留其余改动（因为透明化在 HostingWindow 是一次性的，可快速回退到 `isOpaque=true`）。
